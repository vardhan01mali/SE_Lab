The implementation follows the Object-Oriented Programming (OOP) paradigm and is structured using the Entity-Boundary-Control (EBC) design pattern.  Key FeaturesSecure Authentication: PIN verification with a 3-attempt lockout policy.  Transaction Validation: Checks for minimum withdrawal amounts, daily limits, and multiples of 100.  Automated Testing: Integrated test runners for both Black Box and White Box verification.  📂 File Structurecodecheck.cpp: The primary C++ source code containing the ATM logic and automated test runners.sheet1.csv: Black Box test cases using Equivalence Class Partitioning (ECP) and Boundary Value Analysis (BVA).  whitebox.csv: White Box test cases targeting specific internal logic paths and branch coverage.  ⚙️ Implementation Details (Q2 A.1)The code is divided into three primary components according to the EBC model:  Entity (Account): Stores card details, balance, and account status (Blocked/Active).  Control (BankServer): Contains the core logic for PIN verification and withdrawal validation.  Boundary (ATMInterface): Handles user interactions (simulated via the automated runner for this exam).  🧪 Testing Methodology (Q2 A.2)1. Black Box TestingTechniques: Equivalence Class Partitioning (ECP) and Boundary Value Analysis (BVA).  Coverage: 15 test cases covering valid/invalid PINs, insufficient funds, and limit boundaries.  2. White Box TestingTechniques: Statement Coverage, Branch Coverage, and Path Coverage.
